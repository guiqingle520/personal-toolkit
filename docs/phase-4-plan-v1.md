## 1. Phase 4 规划目标

Phase 4 的目标不是继续补基础闭环，而是在 Phase 3 已完成的能力之上，进入“增强体验 / 扩展协作 / 提升可运营性”的阶段。

当前建议：

- 先做小步快跑的 Sprint 规划
- 继续保持“先计划、后实现、再审查”的执行方式
- 避免一次性引入多个高风险主题并行推进

---

## 2. Phase 4 规划输入

### 2.1 已具备的系统基础

- Todo 基础 CRUD
- 分页 / 筛选 / 回收站
- 子任务 / Checklist
- 重复任务
- 统计面板
- 静态看板视图
- 中英文国际化

### 2.2 当前系统边界

- 看板尚不支持拖拽
- 统计尚不支持钻取和筛选联动
- 当前状态体系仍以 `PENDING / DONE` 为主
- 当前仍是单用户本地工具箱型项目基线

---

## 3. Phase 4 候选方向

基于当前仓库文档与现有能力，建议候选方向如下：

1. **看板增强**
   - 拖拽改状态
   - 更细状态列划分
   - 更强的列内排序

2. **统计增强**
   - 图表化趋势展示
   - 统计卡片联动筛选
   - 更多时间范围

3. **任务增强能力**
   - 提醒 / 通知
   - 更丰富的优先级与到期策略
   - 更强的标签 / 分类管理

4. **工程化增强**
   - 操作手册与 README 完整同步
   - 更完整的 E2E 验证
   - 更强的部署与监控说明

---

## 4. 推荐的下一 Sprint 方向

### 推荐：Sprint 4.1 看板增强

推荐理由：

- Sprint 3.4 已经完成静态看板首版
- 当前最自然的演进方向是看板交互增强，而不是再开全新主题
- 相比从零引入全新子系统，增强现有看板的上下文连续性更强

### Sprint 4.1 候选目标

1. 支持拖拽改状态
2. 在拖拽前明确是否需要扩展状态体系
3. 补看板专属测试
4. 优化看板模式下的交互反馈

---

## 5. Sprint 4.1 预规划范围（候选）

### 5.1 建议纳入

- 看板列间任务拖拽
- 拖拽后状态更新
- 拖拽交互反馈
- 看板专属测试补齐

### 5.2 暂不纳入

- 多维过滤联动重构
- 新图表库引入
- 提醒系统
- 权限与认证体系

---

## 6. 关键决策点

在正式启动 Sprint 4.1 前，需要明确以下问题：

1. 是否继续只保留 `PENDING / DONE` 两态，还是扩展 `IN_PROGRESS`
2. 拖拽是直接落库更新，还是先本地乐观更新再同步
3. 看板是否继续只基于“当前页数据”，还是改为独立拉取整组数据
4. 是否要同步更新统计逻辑以适配未来状态扩展

---

## 7. 风险与约束

### 7.1 当前风险

- 一旦状态体系扩展，将影响列表、统计、看板三处逻辑
- 拖拽会引入更多前端复杂交互和测试成本
- 若缺少明确的状态定义，容易造成交互和统计口径分裂

### 7.2 规划约束

- 必须继续遵守中英文国际化要求
- 必须继续遵守实体类 / 实现方法注释规范
- 必须继续采用文档版本化归档方式

---

## 8. 建议的推进顺序

1. 先确认 Sprint 4.1 是否以“看板拖拽增强”为主线
2. 明确状态体系是否扩展
3. 输出 Sprint 4.1 正式实施方案
4. 再进入实现
